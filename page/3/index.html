apt报错Certificate verification failed的解决方案

发表于 2022-08-11 更新于 2022-08-13

序

今日重新配置WSL2（Debian）开发环境，又遇此问题，故记之。

现象描述

Debian安装好后直接换清华源（使用https协议），再使用apt update时出现上述错误。

问题原因

https协议是需要CA证书的，崭新出厂的Debian没有相应的支持。

解决方案

方案1. 换回之前默认的源，管理员权限下运行指令“apt update && apt install ca-certificates”（大概会很慢，因为官方源不慢那你就不会换源然后遇到这个问题了）
方案2. 将source.list中的所有源地址由https协议更改为http，再运行方案1的指令，最后将http改回https（安全起见）

使用Hexo框架和Github Pages搭建静态博客

发表于 2022-08-10 更新于 2022-09-16

Hexo was originally created and maintained by Tommy Chen in 2012. Since then, it has helped thousands of people to build their dream website/blog.

序

作为一名“Geek”，哪能没有自己的博客？这就好像一名司机必须要有车，而车需要花买，服务器也需要。但好在博客不一定要有自己的服务器。车，也不一定非要是自己的。

2020年的夏天，Hexo框架偶然进入我的视野，这是一个基于Node.js构建的静态博客框架，该框架以Markdown作为博文源码，根据自定义的博客主题自动生成html页面。配合Github pages自动部署，甚至可以做到零成本搭建博客。

时隔两年，恍惚中，我竟再次回到了这里，只愿记录下自己搭建博客的一点心得，为大家献上一篇微不足道的教程。

Hexo官网
Github Pages介绍

一切开始之前的准备

正如前文所述，Hexo基于Node.js构建，考虑到Node.js开发环境在Windows系统上的种种不便利因素，我选择了WSL2（Debian），如果你坚持要在Windows上完成搭建，这篇博文依然适用，不过你可能需要格外找方法在Windows上搭建下述环境（因为我懒）。

如果你不知道什么是WSL，请翻阅以下文档

安装Node.js，npm，以及hexo模块

首先我们需要安装Node.js运行时及其包管理器npm。
[]

Node.js发布于2009年5月，由Ryan Dahl开发，是一个基于Chrome V8引擎的JavaScript运行环境，使用了一个事件驱动、非阻塞式I/O模型，让JavaScript 运行在服务端的开发平台，它让JavaScript成为与PHP、Python、Perl、Ruby等服务端语言平起平坐的脚本语言

在Debian上可以使用命令apt install nodejs来安装nodejs（需要管理员权限），其余发行版请使用相应的包管理器并寻找对应的包名，下同。
Hexo框架并不是Node.js的一部分，它被封装为Node.js的一个第三方模块，需要使用Node.js的包管理器（npm）进行安装。在Debian上可以使用命令apt install npm进行安装。
接着我们需要使用npm安装Hexo框架，使用指令npm install -g hexo-cli即可（需要管理员权限），其中，-g代表全局安装。

搭建Github Pages

由于没有专用的云服务器用于运行博客，我们需要寻找一家可靠的托管方，比如Github。

首先你需要登入你的Github账号，并创建一个新的Repository，其命名格式需为username.gitHub.io（比如我的，iaksh.github.io）。符合该命名格式的Repository会自动进行Github Pages构建。
当然，截至目前，这里可能只有一个孤零零的README.md。

创建你的Hexo博客

找一个你喜欢的地方兴建文件夹，然后在终端中cd进哪个文件夹并使用命令hexo init，一个hexo博客工程就完成了。
此时你可以使用hexo server（或简写为hexo s)试运行博客，使用浏览器访问localhots:4000即可预览。
使用hexo new name（hexo n name）即可开始一篇新的博文，此命令将会在工程根目录的source文件夹中生成对应名称的markdown源文件，该markdown源文件即为你的博文源码。
其他的命令还有hexo generate（hexo g），hexo deploy（hexo d）。前者能将博客生成为一份完整的html+css+js文件，后者自动将其按照预设配置进行部署

修改你的博客配置

工程根目录中有一配置文件名为“_config.yml”，这是hexo最顶层的配置文件，在这里你可以指定网站的一些基本内容。此外，你可以将第三方主题包放在工程根目录的themes文件夹中，然后修改主题包内的“_config.yml”。
另：由于各个主题配置文件不尽相同，具体请见其中的注释。

将你的Hexo博客推送至Github并自动部署

在推送博客至github之前，你需要在工程根目录启用hexo的git部署模块，命令为npm install hexo-deployer-git –save
完成上述操作后，还需修改工程根目录下的“_config.yml”中的deploy项目，将其中的type改为git，添加repo子项以指定git仓库，添加branch子项以指定上传分支。
在这之后，使用指令hexo g && hexo d即可自动完成博客的生成与部署。

可能遇到的问题以及解决方案

我无法用Github密码完成Push

在进行推送时，git会需要你输入自己的Gihub账户名以及密码，但你可能已经发现，密码已经不再适用了。
事实上，自2021年8月13日起，github不再支持使用密码在git上push代码，取而代之的是token。

As previously announced, starting on August 13, 2021, at 09:00 PST, we will no longer accept account passwords when authenticating Git operations on GitHub.com. Instead, token-based authentication (for example, personal access, OAuth, SSH Key, or GitHub App installation token) will be required for all authenticated Git operations. Please refer to this blog post for instructions on what you need to do to continue using git operations securely.
Removal

你需要在自己的github设置中找到“Developer Settings”中的“Personal access tokens”，点击右侧按钮新建token，勾选repo权限，然后填入git的密码框中即可。
需要格外注意的是，token被创建之后仅会被显示一次，你需要自己记好它。

我连接不上Github

这是一个非常令人头痛的问题，种种不稳定因素干扰下，Github在国内可谓是风雨飘摇。但就目前而言，你可以使用github镜像站（比如fastgit）来加速访问。
当然，我不能保证它们是绝对安全的，你需要自己承担风险。

Github Pages没有更新我刚刚Push的内容

你先别急。

Github Pages的部署是需要时间的，具体视Github服务器状况而定，实在着急的话可以去Github仓库的Environment里看一眼Deployment history，还是不行就多等等吧。

另外，Github Pages的服务器运行在UTC时区，而我们国内通常是UTC+8（北京时间），Github Pages只会编译在当前时间（UTC）之前的内容，所以如果你的博文长时间没有被自动部署请尝试将时间更改到一天前。
